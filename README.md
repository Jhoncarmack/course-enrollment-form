## 프로젝트 개요

온라인 교육 플랫폼의 수강 신청 흐름을 다단계 폼으로 구현한 프로젝트다.

사용자는 강의 목록에서 강의를 선택하고, 개인 또는 단체 신청 유형에 따라 정보를 입력한 뒤, 최종 확인
화면에서 내용을 검토하고 제출한다.

신청은 3단계로 구성되며, 각 단계에서 입력한 값은 이전 단계로 돌아가도 유지되도록 설계했다.
단체 신청 시에는 단체명, 인원수, 참가자 명단, 담당자 연락처 등의 추가 필드가 조건부로 표시된다.

본 프로젝트는 프로덕트 엔지니어 채용 과제 FE-A로 작성되었다.

## 기술 스택

- **TypeScript**: 과제 필수 요구사항. 데이터의 모양(이름, 이메일, 강의 정보 등)을 약속처럼
  정의해두면 잘못된 값이 들어왔을 때 바로 잡아준다.

- **Next.js (App Router)**: 프론트엔드 화면과 가짜 API를 한 프로젝트에서 같이 관리할 수 있다.
  `app/api/courses/route.ts` 같은 파일을 만들어두면, 누군가 `/api/courses`로 요청을 보낼 때
  그 파일이 자동으로 실행돼서 응답을 돌려준다. 별도 백엔드 서버 없이 `npm run dev` 하나로
  화면과 API가 같이 동작.

- **React**: Next.js 기반의 UI 라이브러리.

- **TanStack Query**: 서버에서 받아오는 데이터(강의 목록, 신청 제출 등)의 상태를 자동으로
  관리해주는 도구다. '지금 받아오는 중', '에러 났음', '받아온 결과' 같은 상태를 직접 관리할 필요 없이
  가져다 쓰면 된다. 같은 데이터를 또 요청하면 캐시(잠시 보관해 둔 결과)에서 바로 꺼내준다.

- **CSS (글로벌 CSS)**: 별도 UI 라이브러리 없이 작성. 디자인 자유 항목이라 기능과 UX에 집중.

## 실행 방법

### 사전 요구사항

- Node.js 20 이상
- npm

### 설치 및 실행

```bash
git clone https://github.com/Jhoncarmack/course-enrollment-form.git
cd course-enrollment-form
npm install
npm run dev
```

브라우저에서 [http://localhost:3000](http://localhost:3000) 으로 접속합니다.

### 빌드 확인

```bash
npm run build
```

### 폴더별 역할

- **app/**: Next.js App Router 기반 페이지와 Mock API 라우트가 위치한다.
- **components/**: 단계별 화면 컴포넌트와 공통 UI 컴포넌트가 분리되어 있다.
- **lib/**: 화면과 분리된 비즈니스 로직(API 호출, 유효성 검증, 포커스 이동 도우미).
- **types/**: 화면 컴포넌트와 Mock API가 공유하는 타입 정의를 한 곳에 모았다.

## 요구사항 해석 및 가정

#### 1. 단체 신청에서 개인 신청으로 전환 시 데이터 처리

과제 명세에는 "개인 신청과 단체 신청은 입력 필드가 다르다"라고만 적혀 있고, 유형을 바꿀 때 단체 정보를
어떻게 처리할지는 정의되어 있지 않다.

**결정**: 단체 → 개인으로 전환할 때 단체 정보가 입력되어 있으면 확인 대화상자를 띄우고, 사용자가
동의하면 단체 정보를 초기값으로 되돌린다. 비어 있는 상태에서는 확인 없이 즉시 전환한다.

**근거**: 사용자가 실수로 유형을 잘못 눌렀을 때 입력한 단체 정보가 말없이 사라지면 혼란스럽다.
동시에 빈 상태에서까지 매번 확인을 요구하면 불필요한 마찰만 생긴다.

#### 2. 정원이 거의 찬 강의의 표시

과제 명세에는 마감(정원 초과) 강의 처리만 언급되어 있다.

**결정**: 잔여 정원이 3 이하면 "마감 임박" 노란 뱃지를 추가로 표시한다. 선택 자체는 가능하다.

**근거**: 평가 기준에 "정원이 거의 찬 강의를 선택했을 때 UX"가 명시되어 있다. 마감/일반 강의만
구분하면 사용자가 임박 상태를 인지하기 어렵다.

#### 3. 참가자 명단의 이메일 중복 처리

과제 명세에는 참가자 이메일 중복에 대한 처리 방식이 명시되어 있지 않다.

**결정**: 참가자 명단 안에서 같은 이메일이 두 개 이상 등장하면 중복으로 판정하고, 해당 모든 항목에
"참가자 이메일은 중복될 수 없습니다" 메시지를 표시한다. 검증은 다음 단계로 넘어갈 때(전체 검증 시점)에
수행한다.

**근거**: blur 시점에는 한 필드만 검증하므로 다른 참가자와의 비교가 어렵다. 명단 전체를 한 번에
훑어야 하는 검증이라 전체 검증 단계로 일원화했다.

대표자 이메일과 참가자 이메일의 중복은 검증하지 않았다. 명세에 정의되지 않았고, 한 명이 대표자이자
참가자인 경우도 가능하다고 보았다.

#### 4. 한국 전화번호 형식

과제 명세는 "한국 전화번호 형식"이라고만 기술되어 있다.

**결정**: 일반적으로 쓰이는 휴대전화 번호와 지역번호를 모두 허용하기로 했다. 자동 하이픈 포매팅은
적용하지 않았고, 사용자가 `010-1234-5678` 형식으로 직접 입력한다.

**근거**: 어떤 형식을 표준으로 잡을지 별도 기준이 없어, 가능한 형식을 폭넓게 허용했다.
자동 포매팅은 시간상 보류했다.

#### 5. Mock API 시뮬레이션 트리거

평가 시 에러 처리를 직접 확인할 수 있도록 다음 트리거를 두었다.

| 시나리오             | 트리거                                                  | 결과                    |
| -------------------- | ------------------------------------------------------- | ----------------------- |
| COURSE_FULL          | "UX/UI 디자인 입문"(course-3) 선택 후 제출              | 정원 초과 에러          |
| DUPLICATE_ENROLLMENT | 신청자 이메일에 `duplicate@example.com`                 | 중복 신청 에러          |
| INVALID_INPUT        | 약관 미동의, 신청자 필수 정보 누락, 단체 정보 불완전 등 | 입력값 오류             |
| 정상                 | 위 조건을 피해서 제출                                   | 신청 완료 화면으로 이동 |

원래 COURSE_FULL 트리거를 정원이 가득 찬 course-2에 두었으나,
1단계에서 마감 처리되어 사용자가 선택할 수 없는 상태였다.
선택 가능한 course-3로 옮겨 평가 시 시연 가능하게 했다.

INVALID_INPUT 케이스는 정상 화면 흐름에선 클라이언트 검증이 먼저 막기 때문에 사용자에게
노출되지 않는다. 클라이언트 검증을 우회한 요청(예: 개발자 도구로 버튼의 disabled 해제)이
서버에 도달한 경우에만 발생한다.

## 설계 결정과 이유

#### 1. 폼 상태 관리: page.tsx에서 한 곳에 모아서 관리

사용자가 입력하는 값(선택한 강의, 신청 유형, 신청자 정보 등)은 모두 `app/page.tsx` 한 곳에
모아둔다. 각 단계 컴포넌트(`CourseStep`, `ApplicantStep`, `ConfirmStep`)는 자기 안에 값을
저장하지 않는다. page.tsx가 가지고 있는 값을 props로 받아와서 보여주기만 하고, 사용자가
입력하면 page.tsx에 알려서 거기서 값을 바꾸게 한다.

이렇게 한 이유는 단계 사이에서 같은 정보를 공유해야 하기 때문이다. 예를 들어 1단계
`CourseStep`에서 사용자가 "프론트엔드 입문" 강의를 골랐다면, 3단계 `ConfirmStep`
확인 화면에서도 그 강의 이름을 보여줘야 한다. 그런데 강의 정보를 `CourseStep` 안에만
저장해두면 `ConfirmStep`은 그 정보를 알 수가 없다. 그래서 세 컴포넌트의 위쪽인
page.tsx에 정보를 모아두고, 각 컴포넌트가 거기서 정보를 받아간다.

폼 전용 라이브러리(React Hook Form 등)도 있지만, 과제 규모에선
React 기본 기능(useState)만으로 충분했다.

#### 2. Mock API: Next.js Route Handler

별도 도구나 패키지 없이 Next.js가 기본 제공하는 기능으로 가짜 API를 만들었다.
`app/api` 폴더에 파일을 두면 그 파일이 알아서 API 역할을 한다. `npm run dev` 하나로 프론트엔드와
가짜 API가 같이 뜬다.

#### 3. 임시 저장과 이탈 방지

사용자가 폼을 채우는 중에 실수로 새로고침하거나 탭을 닫아도 입력값이 사라지지 않도록 두 장치를 두었다.

- **임시 저장(localStorage):** 입력값이 바뀔 때마다 브라우저의 localStorage에
  자동으로 저장한다. 페이지를 다시 열면 저장된 값을 읽어서 입력칸에 다시 채워준다.
  제출이 성공하면 저장된 데이터는 삭제한다.

- **이탈 방지(beforeunload):** 입력 중에 새로고침이나 탭 닫기를 시도하면 브라우저가
  "정말 떠나시겠습니까?" 경고창을 띄운다. 빈 화면(아직 아무것도 입력 안 함)이거나 제출이 끝났을 때는
  경고를 띄우지 않는다.

#### 4. 클라이언트와 서버 양쪽에서 검증

브라우저(클라이언트)와 가짜 API(서버) 양쪽에서 검증한다.

- 클라이언트 검증은 사용자에게 빠른 피드백을 주기 위해서다. 서버까지 가지 않아도 잘못된 입력이면
  즉시 에러 메시지를 보여준다.

- 서버 검증은 두 가지 이유로 필요하다.
   - 첫째, 클라이언트 검증은 우회될 수 있다. 브라우저 개발자 도구(F12)로 검증 코드를 임의로 끄거나,
     화면을 거치지 않고 curl 같은 도구로 서버에 직접 요청을 보내면 클라이언트 검증은 실행되지 않는다.
     서버에서 한 번 더 검사해야 잘못된 데이터가 들어오는 걸 막을 수 있다.
   - 둘째, 정원 초과나 중복 신청 같은 건 서버(데이터베이스)만 아는 정보라서 서버에서만 검증할 수 있다.

#### 5. 제출 실패 시 입력값 유지

수강 신청 제출이 실패해도 입력한 값은 모두 그대로 남는다. 폼 상태를 page.tsx에 통합 관리한 덕에
자동으로 보존된다(제출 실패는 화면 상태를 건드리지 않으므로).
사용자는 에러 메시지를 보고 다시 "제출" 버튼을 누르면 같은 데이터로 재시도할 수 있다.

비즈니스 에러(정원 초과, 중복 신청, 입력값 오류)는 서버가 주는 에러 코드별로 사용자가 알아듣기
쉬운 메시지로 바꿔서 보여준다. 그 외(네트워크 에러 등 알 수 없는 오류)는
일반 메시지("다시 시도해 주세요")로 안내한다.

#### 6. 유효성 검증: 두 시점, 화면 코드와 분리

검증 규칙(이메일 형식, 전화번호 형식, 필수 입력 등)은 화면 코드에 넣지 않고
`lib/validations.ts`에 따로 모았다. 화면 코드(ApplicantStep 등)는 "blur가 일어났네" 같은
화면 동작만 책임지고, "이 입력이 올바른가?"는 lib의 함수에 물어본다.

이렇게 분리한 이유는 같은 검증 규칙을 두 시점에 모두 써야 하기 때문이다.

- **다음 단계 클릭 시 (전체 검증):** 해당 단계의 모든 필드를 한 번에 검사한다.
- **필드 blur 시 (단일 검증):** 사용자가 한 칸에 입력을 끝내고 다른 칸으로 옮길 때,
  그 한 필드만 검사한다.

검증 규칙이 화면 코드 안에 있으면 두 시점에 같은 규칙을 두 번 적어야 한다. 그러면 나중에 규칙이 바뀔 때 한 곳만 고치고 다른 곳을 빠뜨릴 위험이 있다. lib에 모아두면 한 번만 고쳐도 두 시점 모두 자동으로 반영된다.

입력을 다시 시작하면 그 필드의 에러는 즉시 제거하고, 전체 검증에서 실패한 경우엔 첫 에러 필드로
자동 포커스를 이동한다(`lib/focusError.ts`). 사용자가 한 번 입력하고 한참 뒤에야 에러를 보거나,
어느 칸에 문제가 있는지 직접 찾아야 하는 상황을 막기 위해서다.

참가자 이메일 중복 검사는 명단 전체를 훑어봐야 하므로 blur 시점에서는 제외하고,
전체 검증에서만 수행한다.

## 미구현 / 제약사항

#### 1. 테스트 코드

과제 명세에서 FE는 테스트 코드를 권장 사항으로 명시했다. 시간 분배상 유효성
검증 로직(blur 검증, 전체 검증)과 폼 흐름 테스트를 작성하지 못했다.

추후 추가한다면 우선순위:

- `lib/validations.ts`의 순수 함수 단위 테스트
- 단체→개인 전환 시 데이터 초기화 동작 테스트
- 제출 실패 시 입력값 유지 여부 테스트

#### 2. 인원수 축소 시 입력 데이터 보호

단체 신청에서 인원수를 줄이면(예: 5명 → 2명) 3, 4, 5번 참가자의 입력 데이터가 즉시 사라진다.
단체→개인 전환에는 확인 대화상자를 두었지만, 인원수 축소에는 같은 보호 장치를 두지 않아
일관성이 부족하다. 동일하게 확인 대화상자를 띄우는 방식으로 보완할 수 있다.

#### 3. localStorage 복구 시 강의 정보 재검증

새로고침 후 임시 저장 데이터를 복구할 때, 그 사이 강의 정원이 가득 찼는지 다시 확인하지 않는다.
사용자는 마감된 강의를 들고 진행하다 제출 시점에 비로소 COURSE_FULL 에러를 본다.
복구 시점에 강의 정보를 다시 조회하면 더 자연스럽다.

#### 4. 뒤로가기 이벤트 처리 (이론적 한계)

이탈 방지는 `beforeunload` 이벤트 기반이다. 사용자가 사이트를 뜨려는
동작(다른 주소로 이동, 새로고침, 탭 닫기, 외부 사이트로 뒤로가기)은 모두 잡힌다.

다만 만약 이 앱이 여러 페이지(`/courses`, `/enrollment` 등)로 나뉘어 있다면,
같은 사이트 안에서의 페이지 이동은 `beforeunload`로 못 잡고 별도 처리(Next.js 라우터 이벤트 등)가
필요하다. 현재 프로젝트는 단일 페이지 구조라 이 문제가 실제로 발생하지는 않는다.

## AI 활용 범위

JavaScript 기초 수준에서 시작해 코드 작성 전반에 ChatGPT를 적극 활용했다.
TypeScript, Next.js, TanStack Query는 본 과제에서 처음 다뤘다.

요구사항 해석과 의사 결정, 결과물 검증은 본인이 직접 했고, AI는 코드 작성을 보조하는 도구로 활용했다.

### 본인이 결정한 것

- 단체 신청에서 개인 신청으로 전환 시 단체 데이터 처리 방식 (확인 대화상자 + 초기화, 빈 상태면 즉시 전환)
- 참가자 이메일 중복 검증의 시점을 검토한 결과, blur 단계에서 제외하고
  전체 검증에서만 처리하기로 판단 (단일 필드 검증과 명단 전체 비교 검증은 성격이 달라서)
- 마감 임박 노란 뱃지 추가 (잔여 정원 3 이하 강의 표시)
- COURSE_FULL 에러 시뮬레이션 트리거를
  선택 가능한 강의(course-3)로 변경 (평가 시 에러 처리 시연 가능하도록)
- 푸터를 신청 완료 화면에도 표시 (어느 화면에서든 문의처를 확인할 수 있도록)
- 한국 전화번호 형식: 휴대전화·지역번호 모두 허용, 자동 하이픈 포매팅은 시간상 보류
- 이탈 방지 경고를 모든 상황에서 띄우지 않고, 빈 화면이거나 제출이 끝났을 때는 띄우지 않도록 처리

### 본인이 발견하고 수정한 결함

- 2단계에서 발생한 검증 에러가 1단계로 돌아간 뒤에도 그대로 남아 있던 버그
- 3단계 제출 실패 후 다른 단계로 돌아갔다 와도 에러 메시지가 남던 문제
- COURSE_FULL 트리거 강의가 1단계에서 마감 처리되어 평가 시 시연이 불가능했던 문제
- 띄어쓰기 없는 긴 텍스트 입력 시 textarea와 확인/완료 화면이 가로로 흘러 레이아웃이 깨지던 문제

### 검증 방식

각 기능을 추가할 때마다 `npm run dev`로 직접 시연하며 동작을 확인했다. 의도와 다른 부분은
즉시 수정하고 별도 커밋으로 분리해 변경 이력을 남겼다.

### 한계

- React 훅(useEffect, useState 등) 동작 원리와
  TanStack Query 내부 동작을 완전히 이해하지 못한 채 적용한 부분이 있다.
- TypeScript 고급 패턴(discriminated union 등)은 적용했으나 심화 이해가 부족하다.
- 폼 상태를 스텝별 schema로 분리하거나, 입력 칸을 재사용 가능한 폼 필드 컴포넌트로
  추출하는 단계까지는 가지 않았다. 과제 규모상 통합 useState와 단계 컴포넌트
  직접 작성으로도 충분하다고 판단했다.
- 위 영역은 후속 학습에서 보완할 계획이다.
